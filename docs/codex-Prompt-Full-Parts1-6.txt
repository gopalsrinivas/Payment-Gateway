# Payment Gateway Demo

## Software Requirement Specification (SRS)

Version: 1.0

---

# 1. Project Overview

## Project Name

Payment Gateway Demo using Razorpay Test Mode

## Purpose

The purpose of this project is to develop a small full-stack Payment Gateway Demo application that demonstrates Razorpay payment integration using Test Mode.

The application should allow users to register, log in, view products, add products to a cart, create orders, complete payments using Razorpay Checkout, verify payment signatures, and view order and payment history.

The application should be production-style but interview-friendly, focusing on clean architecture, modular code, authentication, validations, logging, security, REST APIs, payment verification, webhooks, Docker, and CI/CD practices.

Part 1 should create only the project foundation.

Part 1 should not implement complete product, cart, order, checkout, payment verification, payment history, refund, or webhook-processing functionality.

# 7. Payment Flow

The complete payment flow should be:

```text
User Login
    ↓
View Products
    ↓
Add Products to Cart
    ↓
Create Application Order
    ↓
Backend Calculates Trusted Amount
    ↓
Backend Creates Razorpay Order
    ↓
Frontend Opens Razorpay Checkout
    ↓
Customer Completes Payment
    ↓
Razorpay Returns Payment Details
    ↓
Frontend Sends Payment Details to Backend
    ↓
Backend Verifies Payment Signature
    ↓
Backend Updates Payment Status
    ↓
Backend Updates Order Status
    ↓
Payment Success Page
```

Webhook flow:

```text
Razorpay
    ↓
Webhook Endpoint
    ↓
Read Exact Raw Request Body
    ↓
Verify Webhook Signature
    ↓
Store Webhook Event
    ↓
Prevent Duplicate Processing
    ↓
Update Payment Status
    ↓
Update Order Status
```

Part 1 should create only the Razorpay configuration and webhook route foundation.

Part 1 should not create actual Razorpay orders or process payments.

# 8. Authentication Requirements

Part 1 should implement the following APIs:

```text
POST /api/v1/auth/register
POST /api/v1/auth/login
POST /api/v1/auth/logout
GET  /api/v1/auth/profile
```

## Registration

Registration should:

* Validate name
* Validate email
* Validate password
* Require a minimum password length of 8 characters
* Prevent duplicate email registration
* Assign Customer role by default
* Hash the password
* Store the user in PostgreSQL
* Return safe user information
* Never return the password

## Login

Login should:

* Validate email and password
* Verify that the user exists
* Reject inactive users
* Compare the hashed password
* Generate a JWT token
* Return safe user information
* Return role information
* Never log or return the password

## Profile

Profile API should:

* Require a valid JWT
* Return the authenticated user
* Include role information
* Exclude the password

## Logout

JWT authentication is stateless.

Logout should:

* Return a success response from the backend
* Remove the token from the frontend
* Not implement JWT blacklisting in Part 1

# 12. Coding Standards

Every controller should contain only request and response handling.

Business logic should be written inside services.

Database operations should be written using Sequelize models.

Do not create a repository layer for this project.

Every API should have validation.

Every API should return a standard response format.

Controllers should pass errors to centralized error middleware using `next(error)`.

Do not write direct SQL queries inside controllers.

Do not write Sequelize queries directly inside controllers.

Use async/await everywhere.

Use environment variables for configuration.

Follow REST naming conventions.

Use transactions where multiple related database updates are required.

Never log:

* Passwords
* JWT tokens
* Razorpay Key Secret
* Razorpay Webhook Secret
* Complete sensitive request bodies
* Card information

Never trust payment amounts sent by the frontend.

The final payable amount should eventually be calculated using trusted product and order data from the backend.
