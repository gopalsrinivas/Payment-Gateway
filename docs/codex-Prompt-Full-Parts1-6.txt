# Payment Gateway Demo

## Software Requirement Specification (SRS)

Version: 1.0

---

# 1. Project Overview

## Project Name

Payment Gateway Demo using Razorpay Test Mode

## Purpose

The purpose of this project is to develop a small full-stack Payment Gateway Demo application that demonstrates Razorpay payment integration using Test Mode.

The application should allow users to register, log in, view products, add products to a cart, create orders, complete payments using Razorpay Checkout, verify payment signatures, and view order and payment history.

The application should be production-style but interview-friendly, focusing on clean architecture, modular code, authentication, validations, logging, security, REST APIs, payment verification, webhooks, Docker, and CI/CD practices.

Part 1 should create only the project foundation.

Part 1 should not implement complete product, cart, order, checkout, payment verification, payment history, refund, or webhook-processing functionality.

# 7. Payment Flow

The complete payment flow should be:

```text
User Login
    ↓
View Products
    ↓
Add Products to Cart
    ↓
Create Application Order
    ↓
Backend Calculates Trusted Amount
    ↓
Backend Creates Razorpay Order
    ↓
Frontend Opens Razorpay Checkout
    ↓
Customer Completes Payment
    ↓
Razorpay Returns Payment Details
    ↓
Frontend Sends Payment Details to Backend
    ↓
Backend Verifies Payment Signature
    ↓
Backend Updates Payment Status
    ↓
Backend Updates Order Status
    ↓
Payment Success Page
```

Webhook flow:

```text
Razorpay
    ↓
Webhook Endpoint
    ↓
Read Exact Raw Request Body
    ↓
Verify Webhook Signature
    ↓
Store Webhook Event
    ↓
Prevent Duplicate Processing
    ↓
Update Payment Status
    ↓
Update Order Status
```

Part 1 should create only the Razorpay configuration and webhook route foundation.

Part 1 should not create actual Razorpay orders or process payments.

# 8. Authentication Requirements

Part 1 should implement the following APIs:

```text
POST /api/v1/auth/register
POST /api/v1/auth/login
POST /api/v1/auth/logout
GET  /api/v1/auth/profile
```

## Registration

Registration should:

* Validate name
* Validate email
* Validate password
* Require a minimum password length of 8 characters
* Prevent duplicate email registration
* Assign Customer role by default
* Hash the password
* Store the user in PostgreSQL
* Return safe user information
* Never return the password

## Login

Login should:

* Validate email and password
* Verify that the user exists
* Reject inactive users
* Compare the hashed password
* Generate a JWT token
* Return safe user information
* Return role information
* Never log or return the password

## Profile

Profile API should:

* Require a valid JWT
* Return the authenticated user
* Include role information
* Exclude the password

## Logout

JWT authentication is stateless.

Logout should:

* Return a success response from the backend
* Remove the token from the frontend
* Not implement JWT blacklisting in Part 1

# 12. Coding Standards

Every controller should contain only request and response handling.

Business logic should be written inside services.

Database operations should be written using Sequelize models.

Do not create a repository layer for this project.

Every API should have validation.

Every API should return a standard response format.

Controllers should pass errors to centralized error middleware using `next(error)`.

Do not write direct SQL queries inside controllers.

Do not write Sequelize queries directly inside controllers.

Use async/await everywhere.

Use environment variables for configuration.

Follow REST naming conventions.

Use transactions where multiple related database updates are required.

Never log:

* Passwords
* JWT tokens
* Razorpay Key Secret
* Razorpay Webhook Secret
* Complete sensitive request bodies
* Card information

Never trust payment amounts sent by the frontend.

The final payable amount should eventually be calculated using trusted product and order data from the backend.

# Payment Verification API

Endpoint:

```text
POST /api/v1/payments/verify
```

Customer authentication required.

Request body:

```json
{
  "paymentId": 10,
  "razorpayOrderId": "order_xxxxx",
  "razorpayPaymentId": "pay_xxxxx",
  "razorpaySignature": "signature_value"
}
```

Required flow:

```text
Validate request
→ Load local Payment
→ Validate Customer ownership
→ Match stored Razorpay Order ID
→ Build signature payload
→ Verify HMAC SHA256
→ Reject invalid signature
→ Update Payment in transaction
→ Update Order payment status
→ Update Order status where documented
→ Write Payment Log
→ Return verified result
```

Signature payload:

```text
razorpay_order_id + "|" + razorpay_payment_id
```

Generate expected signature using:

```text
HMAC SHA256 with RAZORPAY_KEY_SECRET
```

Compare signatures safely.

Do not use plain insecure comparison where a timing-safe comparison is practical.

---

# Payment Verification Success

On valid signature:

Payment:

```text
status = CAPTURED or AUTHORIZED
razorpay_payment_id = received payment ID
verified_at = current timestamp
paid_at = current timestamp where appropriate
updated_by = authenticated user ID
```

Order:

```text
payment_status = PAID
```

Order status should change only according to documented business rules.

Possible safe update:

```text
PENDING → CONFIRMED
```

Do not overwrite a later valid Admin status.

Write Payment Log:

```text
event_type = PAYMENT_VERIFIED
previous_status
new_status
request_id
safe metadata
```

Return only verified safe payment details.

---

* Update Payment status only if documented.
* Write a failure Payment Log.
* Return HTTP 400 or 401 according to project conventions.
* Do not expose expected signature.
* Do not expose Key Secret.
* Log safe diagnostic information only.

# Webhook Status Synchronization

Examples:

## `payment.authorized`

* Set Payment to AUTHORIZED if current transition is valid.
* Do not move backward from CAPTURED.
* Write log.

## `payment.captured`

* Set Payment to CAPTURED.
* Set Order payment status to PAID.
* Confirm Order where appropriate.
* Write log.

## `payment.failed`

* Set Payment to FAILED if not already successful.
* Do not overwrite CAPTURED with FAILED.
* Update Order payment status only where valid.
* Write log.

## `order.paid`

* Reconcile local Order and Payment.
* Mark Order payment status PAID.
* Do not create duplicate Payment if one already matches.
* Write log.

All transitions must be idempotent.

# Payment Status Transition Rules

Define allowed transitions.

Example:

```text
CREATED → PENDING
PENDING → AUTHORIZED
PENDING → FAILED
AUTHORIZED → CAPTURED
AUTHORIZED → FAILED only if valid
CAPTURED → REFUNDED in future phase only
```

Do not allow:

```text
CAPTURED → FAILED
PAID Order → PENDING payment status
```

Use a shared transition helper.

Backend is the final authority.

---

# Order Payment Status Rules

Example:

```text
PENDING → AUTHORIZED
PENDING → PAID
PENDING → FAILED
FAILED → PENDING for a new retry attempt where documented
AUTHORIZED → PAID
PAID remains PAID
```

Do not downgrade PAID.

Multiple failed attempts should not erase a later successful attempt.

Order payment status should reflect the final effective payment state.
